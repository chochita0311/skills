# Screen Alignment Checklist

Use this checklist before claiming parity work is complete.

## Mode

- mode was selected explicitly: `match`, `adapt`, `extend`, or `reframe`
- mode remained valid after comparing constitution, target, and current screen
- authority order used for the pass matched the selected mode

## Inputs

- current screen was inspected as rendered output
- constitution or durable design rules were read when available
- user-provided screenshots/redlines were incorporated when available
- target screen was inspected for `match`
- target plus preservation constraints were inspected for `adapt`
- related family screens were inspected for `extend`
- target-versus-constitution conflict was stated explicitly for `reframe`

## Comparison Discipline

- mismatch or consistency list was concrete and visual
- issues were classified into narrow categories
- edits were applied in owning layers
- rendered verification happened after each narrow pass

## Design Integrity

- `match`: result reads materially closer to target
- `adapt`: directional borrowing improved alignment without violating constitution
- `adapt`: user-frozen regions stayed literally preserved or were excluded from the adaptation deliverable
- `adapt`: no target-only controls or metadata were added unless backed by current repo data
- `adapt`: no target-only interactions, workflows, or stateful controls were added unless backed by current repo behavior
- `adapt`: target sketch copy did not replace real current-repo content unless that remapping was explicitly requested
- `extend`: new or revised areas read native to existing system
- `reframe`: task was explicitly reframed instead of half-implemented
- no constitutional rule was violated silently
- no redesign was introduced under cleanup wording
- constitution-critical controls (for example search and primary navigation) remained discoverable
- no AI-assistant/profile framing cues were introduced unless constitution explicitly allowed them

## Common Failure Checks

- topbar rhythm did not drift from target/family pattern
- typography did not become unnecessarily heavier, darker, or louder
- controls did not become rounder, boxier, or more elevated than target/family
- borders and shadows did not become louder than target/family
- layout widths did not create dead zones or compression
- remaining differences were not just cache artifacts
- no foreign component language was introduced during extension work
- no structurally foreign target was treated as ordinary parity work
- directional targets were not copied more literally than preservation constraints allow
- sketch-only fields such as sort controls, read times, counters, or utility metadata were not imported without current-repo support
- sketch-only workflows such as filters, menus, bulk actions, saved views, pagination, or hover tools were not imported without current-repo support

## Data Versus Style

- remaining differences were checked as content-driven vs style-driven
- data-density differences were not misdiagnosed as style errors
- local/inline prototype overrides were identified explicitly
- target-only sketch fields were separated from real current-repo fields before adaptation
- target-only sketch behaviors were separated from real current-repo behavior before adaptation
- target placeholder or fabricated content was separated from real current-repo content before adaptation

## Closure

- remaining differences are minor or intentionally accepted
- temporary screenshots, metrics, and parity artifacts were removed
- temporary local overrides were removed or explicitly documented as temporary
- final result is defensible against constitution, target/family references, preservation constraints, and direct human visual comparison
- if parity could not be completed honestly, that was stated explicitly
